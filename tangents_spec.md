# Tangents: Application Design & Technical Specification

## 1. Product Overview
**Tangents** is an AI chat application designed to solve the "linear constraint" of traditional chatbots. It combines standard chat interfaces with Git-like history tracking, allowing users to branch conversations into divergent history lines to explore side thoughts, swap AI models contextually, and synthesize side-threads back into the main conversation without polluting the primary context window.

## 2. UI Layout

The application uses a **dual-pane desktop layout** (not responsive for mobile at this stage; a mobile layout may be considered in a future version):

* **Left pane** — A vertical sidebar with two switchable modes:
  * **Chats tab**: Lists all stored chats, ordered top (oldest) to bottom (newest). Selecting a chat loads it in the right pane.
  * **Graph tab**: Displays the React Flow node graph for the currently active chat. The main line runs vertically (oldest at top, newest at bottom). All branch nodes extend **down and to the right** of their origin node — there is no leftward branching. Selecting a node in the graph switches the right pane to that node's context.
* **Right pane** — The active chat interface. Displays the linear history of the currently selected branch/node and contains the message input. The input bar contains (left to right): the text field, a **model picker dropdown**, and the **Send** or **Branch** button. When the selected node is not the HEAD of its branch, Send is replaced by Branch. If no API providers are configured, the model picker and Send button are replaced with a prompt directing the user to the Settings page.
* **Settings page** — A dedicated full-page settings view (accessible from a persistent icon in the sidebar). Contains:
  * **Model sources** — add/edit/delete providers, refresh model lists. Includes API key management (write-only per source).
  * **Default model** — the model pre-selected in the picker when starting a new chat.
  * **Synthesis model** — optional override for the model used during merge summarisation and context compression.
  * **Context budget** — per-model-source token limit, pre-populated with known values where available, defaulting to 4,096 for unknown models.
  * **Active share links** — a list of all currently active share URLs generated by the user, with the ability to view and permanently revoke/delete them.
  * **Share link view mode** — controls what visitors see when opening a share link: `"linear"` (default — just the chat history for the shared node, no sidebar or graph) or `"full"` (full app shell in read-only mode, full tree explorable).
  * **Branch naming mode** — select between AI-generated branch titles (using the designated synthesis model) or randomly generated names (e.g., `wispy-river-42`).
  * **Custom instructions** — a global system prompt automatically prepended to the context for all general messages.
  * **Theme** — light or dark (default: **dark**).
  * **Keybindings** — configurable keyboard shortcuts. Built-in defaults:

    | Action | Default |
    | :--- | :--- |
    | Send message | `Enter` (`Shift+Enter` for newline) |
    | New chat | `Ctrl/Cmd+N` |
    | Open Settings | `Ctrl/Cmd+,` |
    | Move to child node (down) | `Ctrl+J` |
    | Move to parent node (up) | `Ctrl+K` |
    | Move into branch (down-right) | `Ctrl+L` |

  The model picker throughout the app only lists models from providers that have been configured.

## 3. Architecture & Tech Stack
* **Backend:** Python + FastAPI (Async, auto-docs, Pydantic validation).
* **Frontend:** React + React Flow (for the interactive node graph).
* **Database:** SQLite (default, zero-config) via **SQLAlchemy** ORM. The abstraction layer allows a future swap to PostgreSQL by setting the `DATABASE_URL` environment variable, with no application code changes required.
* **AI Routing:** LiteLLM (Hot-swappable models: OpenAI, Anthropic, Gemini, local Ollama).
* **Encryption:** `cryptography` (Fernet) for API key encryption at rest. Requires `ENCRYPTION_KEY` env var set by the operator.
* **DevOps & Testing:** * Package management: `uv` by Astral.
  * Test running: `tox` and `tox-uv`.
  * Code quality: `ruff` (linting/formatting), `mypy` (typing).
  * Backend Tests: `pytest`.
  * Frontend Tests: `Vitest` (Unit) and `Playwright` (E2E).
* **Deployment:** **Initial target is single-user self-hosted** with basic authentication (default credentials: `admin` / `tangents`). The architecture is designed to expand to multi-user SaaS via JWT auth and per-user data isolation without structural changes — controlled by the `AUTH_MODE` environment variable.

## 4. Database Schema

> **Engine:** SQLAlchemy ORM. Default engine is SQLite (`DATABASE_URL=sqlite+aiosqlite:///./tangents.db`). To ensure proper cascade deletions in SQLite, foreign keys are strictly enforced at the connection level (`PRAGMA foreign_keys = ON;`). PostgreSQL is supported by setting `DATABASE_URL` to a `postgresql+asyncpg://...` connection string — no application code changes required to switch.

> **Branching UX:** Branching is always **explicit**. Users may branch from the current HEAD or from any prior node. When viewing a node that is not the branch HEAD, the chat input replaces the "Send" button with a **"Branch"** button, making non-HEAD nodes visually read-only. According to user preferences, new branches are given either a randomly generated human-readable name (e.g., `wispy-river-42`) or an AI-generated concise name based on the branch's initial prompt (using the Synthesis model by default). Users can manually rename a branch at any time.

> **Branch lifecycle:** After a merge, the source branch remains alive and usable unless explicitly deleted by the user. There is no special protection on any branch, including `main` — it can be renamed or deleted like any other branch.

> **Chat titles:** Titles are **AI-generated asynchronously**. When the first node of a chat is created, a background task fires a lightweight LLM call to generate a short descriptive title. The sidebar updates the title in-place once the response arrives. Users can manually override the title at any time.

> **Model selection:** Model is chosen **per-node** by the user at send time. The model picker is an inline dropdown positioned immediately to the left of the Send/Branch button in the chat input bar. The previously used model is pre-selected as a default for convenience. The selected model is recorded in `nodes.model_used`.

> **Branch/node deletion:** Deleting a branch triggers a **cascade delete** of all nodes that are exclusively reachable through it and not referenced by any other branch. This cascade applies recursively — all downstream descendants of those nodes are also deleted. The backend must compute the set of exclusively-owned nodes before deletion to avoid removing nodes shared with other branches.

> **Navigation state:** Active chat, branch, and node state is managed entirely in **React state** — not reflected in the URL — for a smooth, SPA-like experience without page navigations. To allow sharing, a user can generate a **share link** via a share button. This creates a `share_links` DB record mapping a short unique token (a full UUID — security through obscurity) to the current `chat_id`, `branch_id`, and `node_id`. Share links are **publicly accessible and read-only** — no authentication is required to view them. Visiting the share URL loads the app in a read-only view of the referenced chat state.

> **New chat creation:** Chats are created **lazily** — no DB record is written when the user clicks "New Chat." The UI switches to a blank compose view immediately, and the chat, its initial `main` branch, and the first node are all created atomically on first message submit.

> **Model sources:** Users configure an arbitrary number of AI sources in Settings. Well-known providers (OpenAI, Anthropic, Gemini, Mistral, etc.) have preset config templates (API key field + optional base URL). Users can also add **custom sources** with a name, base URL, and optional API key (for local endpoints like Ollama, no key is needed). When a source is saved, the backend **immediately fetches the available model list** from that provider and caches it in the `model_source_models` table. Users can manually trigger a refresh of this list from the Settings page. For custom sources that do not expose a model-list endpoint, the user can enter model identifiers manually. The model picker only lists models from configured sources; if none exist, the picker is replaced with a link to Settings.

> **Context window management:** When the linear history of a branch exceeds the active model's configured token budget, the backend automatically **summarises the oldest portion** of the conversation using the same LiteLLM synthesis mechanism as merges. The summary is injected as a synthetic system message at the start of the context sent to the LLM; original nodes are retained in the DB unchanged. Users are notified in the chat UI when context compression has occurred, and a UI element allows them to **review and manually edit the AI-generated summary** to ensure crucial details (like code snippets) are not lost. The token budget is configured **per model source** in Settings — known models are pre-populated with accurate limits, and unknown/custom models default to **4,096 tokens**.

> **Graph node interaction:** Clicking any node in the graph always loads the **full linear history up to and including that node** in the right-pane chat view, reconstructed by walking ancestry via the recursive CTE. Every node has a single unambiguous ancestry chain, so there is no branch ambiguity. A branch-origin node (one with multiple children) shows its own prompt/response plus all ancestor nodes in chronological order — it does not show any of the divergent branches that forked from it. If the clicked node is not the HEAD of any branch, the read-only Branch button is shown.

> **Model source storage:** Source configs are stored in the `model_sources` table. API keys within source configs are **encrypted at rest** using application-level Fernet symmetric encryption (`cryptography` library), keyed from the `ENCRYPTION_KEY` env var. Keys are write-only from the UI — they cannot be read back once submitted, only replaced or deleted. Operators are responsible for securing deployment environment and the `ENCRYPTION_KEY` secret.

> **Streaming:** AI responses are streamed token-by-token to the client via **Server-Sent Events (SSE)**. The `ai_response` column is only written to the database once the full response is received — partial responses are never persisted. If the stream is interrupted (network drop, provider timeout), the partial response is discarded, an error is surfaced to the user, and a **Retry** button is shown to re-send the same prompt.

> **Synthesis model:** The model used for merge synthesis and context compression is resolved in priority order: (1) `synthesis_model_override` in the request payload, (2) `user_settings.synthesis_model_id` if set, (3) the `SYNTHESIS_MODEL` env var if set, (4) the user's currently active model. This lets users configure a cheaper default summarisation model without affecting per-conversation model choices.

The system relies on an **Adjacency List** model to track non-linear history.

## 6. Core Engine Mechanics (Code Snippets)

### 6.1 Linear History Retrieval (The Recursive CTE)
When a user clicks on a tangent branch, the active chat window needs to display the exact linear history leading up to that point. We use a SQLite Recursive CTE to walk backward from the branch's `head_node_id` to the root.

```sql
-- SQLite Query: Fetch linear history for a specific node
WITH RECURSIVE chat_history AS (
    -- Base case: The specific node we are looking at
    SELECT id, parent_id, user_prompt, ai_response, model_used, created_at
    FROM nodes
    WHERE id = :head_node_id
    
    UNION ALL
    
    -- Recursive step: Walk up the tree via parent_id
    SELECT n.id, n.parent_id, n.user_prompt, n.ai_response, n.model_used, n.created_at
    FROM nodes n
    INNER JOIN chat_history ch ON n.id = ch.parent_id
)
-- The CTE builds from bottom to top, so we reverse it to chronological order
SELECT * FROM chat_history ORDER BY created_at ASC;
```

### 6.2 React Flow Graph Mapping (Frontend)
To render the visual Git tree, the backend sends a flat list of all nodes in a chat. The frontend formats this into `nodes` and `edges` for React Flow. The main line runs **vertically** (oldest at top, newest at bottom). All branch nodes extend **down and to the right** of their origin — there is no leftward branching. `dagre` is used for auto-positioning with a top-to-bottom layout direction (`TB`), and branch sub-trees are offset to the right of their parent chain.

All nodes render as **unlabelled selectable dots** — no prompt text or model name is shown statically in the graph view. However, **hovering over a node displays a tooltip with its branch name**, helping users locate specific tangents (e.g., `wispy-river-42`). Individual nodes do not have unique names. Clicking a dot loads that node's context in the right-pane chat view. A node where one or more branches diverge (i.e. it has multiple children) is a **branch origin** and is visually distinguished — e.g. a larger dot or accent colour — but still relies on hover tooltips for branch identification.

```javascript
// Conceptual Frontend logic (React)
const generateGraphData = (dbNodes) => {
  const nodes = [];
  const edges = [];

  // Pre-compute which nodes are branch origins (more than one child)
  const childCounts = {};
  dbNodes.forEach(node => {
    if (node.parent_id) childCounts[node.parent_id] = (childCounts[node.parent_id] || 0) + 1;
  });

  dbNodes.forEach(node => {
    // 1. Create the React Flow Node as a plain dot
    const isBranchOrigin = (childCounts[node.id] || 0) > 1;
    nodes.push({
      id: node.id,
      data: { isBranchOrigin },
      position: { x: 0, y: 0 }, // Auto-layout via Dagre
      type: 'commitDotNode' // Small dot; larger/accented when isBranchOrigin
    });

    // 2. Create the Edge linking to the parent
    if (node.parent_id) {
      edges.push({
        id: `e-${node.parent_id}-${node.id}`,
        source: node.parent_id,
        target: node.id,
        animated: true, // Cool UI effect for active paths
        style: { stroke: '#a8a8a8' }
      });
    }
  });

  return { nodes, edges };
};
```

### 6.3 The "Synthesized Merge" Concept (Backend)
After a merge, the **source branch remains active** — it is not deleted or archived. Users must explicitly delete a branch if they no longer need it.
Instead of dumping raw tangent messages into the main branch, Tangents uses an AI summarization step to "squash and merge" context cleanly.

```python
# Conceptual FastAPI Endpoint Logic
@app.post("/api/chats/{chat_id}/merge")
async def synthesize_and_merge(chat_id: UUID, payload: MergeRequest, db: Session = Depends(get_db)):
    # 1. Fetch full linear history of the source branch using the CTE (above)
    tangent_history = fetch_linear_history(db, payload.source_branch_head_id)
    
    # 2. Format it into a single context string
    raw_context = format_history_to_string(tangent_history)
    
    # 3. Call LiteLLM to synthesize
    synthesis_prompt = f"""
    We explored a side-tangent in our conversation. 
    Here is the raw tangent history: {raw_context}
    Summarize the key findings, code, or decisions made in this tangent so we can 
    continue our main conversation seamlessly.
    """
    ai_summary = litellm.completion(
        # Model priority: payload override → SYNTHESIS_MODEL env var → user's active model
        model=(payload.synthesis_model_override or os.getenv("SYNTHESIS_MODEL") or payload.active_model),
        messages=[{"role": "user", "content": synthesis_prompt}]
    )
    
    # 4. Create a new "Commit" on the target branch (Main) with the summarized findings
    new_node = Node(
        chat_id=chat_id,
        parent_id=get_head_id(db, payload.target_branch),
        user_prompt=f"[Merged Tangent: {payload.source_branch}]",
        ai_response=ai_summary.choices[0].message.content
    )
    db.add(new_node)
    
    # 5. Update Target Branch head pointer
    update_branch_head(db, payload.target_branch, new_node.id)
    db.commit()
    return {"status": "merged", "new_node_id": new_node.id}
```

## 7. Authentication (FastAPI Dependency Injection)
The default mode is **single-user** (`AUTH_MODE=basic`): basic authentication protects the instance (defaults to `admin` / `tangents`), so it's not publicly exposed. When `AUTH_MODE=strict`, the same dependency validates a JWT token, enabling full multi-user SaaS with per-user data isolation. No route code changes are needed to switch modes — only the environment variable.

```python
import os
import secrets
from fastapi import Depends, HTTPException, status
from fastapi.security import HTTPBasic, HTTPBasicCredentials
from jose import jwt, JWTError

AUTH_MODE = os.getenv("AUTH_MODE", "basic") # 'basic' or 'strict'

security_basic = HTTPBasic()

async def get_current_user(
    credentials: HTTPBasicCredentials = Depends(security_basic)
):
    if AUTH_MODE == "basic":
        # Self-hosted mode: Validate against local env var password, default 'tangents'
        correct_username = secrets.compare_digest(credentials.username, os.getenv("ADMIN_USERNAME", "admin"))
        correct_password = secrets.compare_digest(credentials.password, os.getenv("ADMIN_PASSWORD", "tangents"))
        
        if not (correct_username and correct_password):
            raise HTTPException(
                status_code=status.HTTP_401_UNAUTHORIZED,
                detail="Incorrect username or password",
                headers={"WWW-Authenticate": "Basic"},
            )
        return {"id": "00000000-0000-0000-0000-000000000000", "username": credentials.username}
    
    # SaaS Mode: Verify JWT token
    try:
        # Assumes `credentials.password` holds the Bearer token for strict mode.
        payload = jwt.decode(credentials.password, SECRET_KEY, algorithms=[ALGORITHM])
        return {"id": payload.get("sub"), "username": payload.get("username")}
    except JWTError:
        raise HTTPException(status_code=401, detail="Invalid credentials")
```

## 8. Implementation Roadmap

Given the complexity of the application, development should be phased sequentially to ensure core dependencies are stable before adding UI and AI complexities.

### Phase 1: Project Scaffolding & Foundation
1. **Repository Setup:** Initialize the monolithic repository (or separate frontend/backend dirs). Set up `uv` for Python dependency management and Vite for the React frontend.
2. **Database Initialization:** Setup SQLAlchemy with SQLite, making sure to explicitly enable `PRAGMA foreign_keys = ON`. Configure Alembic for running migrations.
3. **Core Schema Implementation:** Create SQLAlchemy models and matching Pydantic DTOs for `Users`, `Settings`, `ModelSources`, and `ModelSourceModels`.
4. **Authentication Base:** Implement the FastAPI dependency injection for `AUTH_MODE=basic` (local admin logic) to secure the API from the start.

### Phase 2: AI & LLM Connectivity
1. **Cryptography & Keys:** Implement the Fernet encryption/decryption module for securely storing API keys.
2. **Settings API:** Create CRUD endpoints for user settings, model sources, and custom instructions.
3. **LiteLLM Integration:** Write the service layer to interact with LiteLLM. Implement the background job/endpoint to fetch and sync available models from configured providers into the database.
4. **Streaming Prototype:** Build a lightweight SSE (Server-Sent Events) wrapper endpoint to test basic chat streaming with LiteLLM before introducing branches.

### Phase 3: The Tangent Engine (Backend Chat Logic)
1. **Chat & Node Schema:** Implement the Adjacency List tables (`chats`, `nodes`, `branches`).
2. **History Retrieval:** Write and optimize the Recursive CTE raw SQL query to reliably fetch linear history for any given `node_id`.
3. **Message Insertion:** Create the endpoints to handle new messages: lazily creating a chat/branch if needed, saving the user prompt, calculating the context window, and streaming the AI response back.
4. **Context Overflows & Synthesis:** Implement the automatic oldest-node summarization logic for context limits, as well as the explicit manual "merge" endpoint.

### Phase 4: Frontend Chat Interface
1. **Base App Shell:** Build the dual-pane layout, theming provider (Dark/Light), and navigation shell.
2. **Settings UI:** Build the settings pages (Model Sources, Defaults, Custom Instructions, Tokens, Link Management).
3. **Chat View:** Implement the linear chat history component. Build the text input with the inline Model Picker dropdown.
4. **SSE Consumer:** Implement the frontend receiver for the Server-Sent Events to render AI responses dynamically as they stream in.

### Phase 5: The React Flow Graph & Branching UI
1. **Graph Data Transformation:** Write the frontend/backend handlers to convert a flat list of chat nodes into React Flow nodes/edges.
2. **Graph Rendering:** Implement `dagre` for the top-to-bottom, right-branching node layout.
3. **Graph Interactivity:** Render nodes as plain dots, build the hover tooltips for branch names, and add the `onClick` event to switch the active chat view to the clicked node's timeline.
4. **Branching UX:** Wire up the UI logic that switches the "Send" button to "Branch" when viewing non-HEAD nodes. 

### Phase 6: Polish & Advanced Features
1. **Async Chat Titles:** Implement the background LLM call to automatically name chats after the first message.
2. **Share Links:** Create the `share_links` table, the generation endpoint, and the read-only frontend view (supporting both `"linear"` and `"full"` modes).
3. **Keybindings:** Add global keyboard event listeners (e.g., `Ctrl+K/J/L` to navigate the graph).
4. **Testing & DevOps:** Finalize Pytest suite, Vitest UI components, Playwright E2E tests, and set up the production `tox` runners. Document the standard deployment process (e.g., Docker Compose).
